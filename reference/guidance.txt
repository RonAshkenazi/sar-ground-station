# Smart Flight Guidance Specification

## Ground Station Refactor — Pre-Localization Adaptive Scanning Layer

## 1. Purpose

The goal of this feature is to add a **Smart Flight Guidance** layer to the Ground Station.

This feature helps the pilot scan the search area more intelligently during the **initial data collection phase**, before the heavy processing pipeline has enough data to run full calibration, enrichment, Re-ID, localization, or final heatmap generation.

The feature should recommend where the scanning unit should fly next in near real time, using only lightweight streaming information received from the Raspberry Pi airborne unit.

The guidance should help achieve three practical goals:

1. Finish the scan faster than a fixed/lawnmower scan.
2. Reduce uncertainty over the search area.
3. Collect more useful samples around areas that appear to contain RF peaks / possible emitters.

This feature is not intended to replace the existing localization system. It is an online data-acquisition optimization layer.

---

# 2. Critical System Boundary

## 2.1 Existing Pipeline Remains Unchanged

The following modules must remain unchanged:

* Calibration
* Enrichment
* Re-ID
* Localization
* Final probabilistic heatmap generation
* Result Analysis
* Saved sessions / final artifact flow

These modules may still run exactly as they currently do after sufficient data has been collected.

## 2.2 New Feature Scope

The new feature is only:

```text
Online Smart Flight Guidance
```

It operates during the live scan and answers:

```text
Where should the pilot fly next to collect more informative data?
```

It must not depend on final localization outputs.

## 2.3 Important Correction

The guidance layer must **not** assume access to:

* Final localized emitter points
* Re-ID clusters
* Post-enrichment data
* Final heatmaps
* Full processed pipeline outputs

Those outputs are not available during the first scan loop.

The guidance layer can only use lightweight streaming data such as:

* Current position
* Time
* Cell ID
* RSSI summaries
* Packet/frame counts
* Dwell time per grid cell
* Basic scan health

---

# 3. Architecture Overview

## 3.1 Recommended Architecture

The Ground Station should become the main active server and processing brain.

The Raspberry Pi should behave as a lightweight airborne sensing client.

```text
Raspberry Pi Air Unit
    |
    | Wi-Fi link
    v
Ground Station Server
    |
    v
Smart Flight Guidance Module
    |
    v
Pilot UI: Map + Grid + Current Target Recommendation
```

## 3.2 Raspberry Pi Responsibilities

The Pi should do only lightweight real-time work:

* Passive Wi-Fi capture
* GPS/pose tagging
* Local raw data logging
* Simple per-window aggregation
* Send small guidance packets to Ground Station
* Maintain local buffering if connection temporarily drops
* Avoid expensive computation

The Pi should not:

* Run localization
* Run Re-ID
* Compute final heatmaps
* Select the next target cell
* Run active sensing optimization
* Stream full CSV data in real time

## 3.3 Ground Station Responsibilities

The Ground Station should:

* Maintain the search-area grid
* Receive pose and evidence packets
* Update per-cell live state
* Compute guidance scores
* Select the next recommended target cell
* Display the recommendation to the pilot
* Log online guidance history for later comparison
* Allow smart-flight vs dumb-flight evaluation

# 4. Conceptual Flow

## 4.1 Before Scan

The user defines:

* Search area boundary
* Grid resolution
* Flight mode:

  * Dumb / fixed scan
  * Smart guidance scan
* Optional scan parameters:

  * RSSI strong threshold
  * Decision interval
  * Grid cell size
  * Weights for scoring function

## 4.2 During Scan

The Raspberry Pi sends lightweight packets:

1. `PosePacket`
2. `EvidencePacket`
3. optional `HealthPacket`

The Ground Station updates:

* Current drone position
* Covered cells
* Evidence map
* Uncertainty map
* Peakness map
* Recommended target cell

The UI displays:

* Current drone location
* Grid overlay
* Current target cell
* Direction and distance to target
* Explore/Refine mode
* Optional score explanation

## 4.3 After Scan

The existing pipeline continues as usual:

```text
Raw Scan Data → Calibration → Enrichment → Re-ID → Localization → Result Analysis
```

The guidance feature adds a separate log:

```text
guidance_history.json / csv
```

This enables comparison between smart and dumb flight.

## 6.2 EvidencePacket — Required

The Pi sends this every **1–2 seconds**.

This packet contains lightweight summary information from a short time window.

Recommended schema:

```ts
EvidencePacketV1 {
  msg_type: "EVIDENCE";
  seq: number;

  t_start_ms: number;
  t_end_ms: number;
  win_ms: number;

  records: CellEvidenceRecord[];
}
```

Each record:

```ts
CellEvidenceRecord {
  cell_id: number;

  dwell_ms: number;

  frames_total: number;
  frames_strong: number;

  rssi_max_dbm: number;
  rssi_p95_dbm: number;
  rssi_mean_dbm?: number;
}
```

Minimum useful record:

```ts
{
  cell_id,
  dwell_ms,
  frames_total,
  frames_strong,
  rssi_max_dbm,
  rssi_p95_dbm
}
```

Optional:

```ts
rssi_mean_dbm
```

## Why These Fields Are Needed

| Field           | Why it matters                                        |
| --------------- | ----------------------------------------------------- |
| `cell_id`       | Maps evidence directly to grid                        |
| `dwell_ms`      | Updates coverage                                      |
| `frames_total`  | Measures packet density                               |
| `frames_strong` | Detects useful high-signal regions                    |
| `rssi_max_dbm`  | Captures possible strong peak events                  |
| `rssi_p95_dbm`  | More stable than max, better than mean                |
| `rssi_mean_dbm` | Helps distinguish weak-dense vs strong-sparse regions |

Do not send the following during the live guidance loop:

* Full CSV
* Full PCAP
* Every raw frame
* Full MAC list
* Full Radiotap records
* Full frame body
* Re-ID features
* Enrichment features
* Heavy JSON arrays without bounds

These should stay local on the Pi and be processed later by the existing pipeline.

# 8. Ground Station Live State

The Ground Station maintains a live state per cell.

For each grid cell (i):

[
s_i = (E_i, U_i, P_i, V_i, A_i, D_i, R_i, J_i)
]

Where:

| Symbol | Meaning                     |
| ------ | --------------------------- |
| (E_i)  | Evidence score              |
| (U_i)  | Operational uncertainty     |
| (P_i)  | Peakness / local contrast   |
| (V_i)  | Coverage / visit score      |
| (A_i)  | Age / staleness             |
| (D_i)  | Travel cost                 |
| (R_i)  | Oscillation/revisit penalty |
| (J_i)  | Final decision score        |

# 9. Parameter Calculations

## 9.1 Evidence Score (E_i)

Evidence measures how interesting the cell appears from raw streaming observations.

It is based on:

* RSSI strength
* Number of frames
* Number of strong frames

### RSSI Normalization

Use a fixed range:

```text
RSSI_MIN = -90 dBm
RSSI_MAX = -55 dBm
```

Normalize:

[
norm(rssi) = clip\left(\frac{rssi - RSSI_{MIN}}{RSSI_{MAX} - RSSI_{MIN}}, 0, 1\right)
]

Example:

| RSSI    | Normalized |
| ------- | ---------- |
| -90 dBm | 0          |
| -80 dBm | 0.286      |
| -70 dBm | 0.571      |
| -60 dBm | 0.857      |
| -55 dBm | 1          |

### Count Score

[
f_{count}(i)=\frac{\log(1+n_i)}{\log(1+n_{ref})}
]

Where:

* (n_i) = `frames_total`
* (n_{ref}) = expected useful frame count per window

Suggested:

```text
n_ref = 30
```

Clamp to [0,1].

### Strong Count Score

[
f_{strong}(i)=\frac{\log(1+n_i^{strong})}{\log(1+n_{strong_ref})}
]

Suggested:

```text
n_strong_ref = 10
```

### RSSI Score

[
f_{rssi}(i)=0.6 \cdot norm(rssi_{p95}) + 0.4 \cdot norm(rssi_{max})
]

### Raw Evidence

[
E_i^{raw}=0.5f_{rssi}+0.3f_{strong}+0.2f_{count}
]

### Time Smoothing

Use exponential moving average:

[
E_i \leftarrow (1-\beta_E)E_i+\beta_EE_i^{raw}
]

Suggested:

```text
beta_E = 0.3
```

This prevents a single RSSI spike from dominating the guidance.

---

## 9.2 Coverage Score (V_i)

Coverage measures how much the system has actually sampled a cell.

[
V_i \leftarrow min\left(1, V_i + \frac{dwell_i}{T_{cov}}\right)
]

Where:

* `dwell_i` = `dwell_ms`
* (T_{cov}) = required dwell time to consider a cell sufficiently covered

Suggested:

```text
T_cov = 5000 ms
```

Interpretation:

| (V_i) | Meaning              |
| ----- | -------------------- |
| 0     | Not covered          |
| 0.5   | Partially covered    |
| 1     | Sufficiently covered |

---

## 9.3 Age / Staleness Score (A_i)

Age measures how stale the information is.

If a cell was updated in the current window:

[
A_i = 0
]

Otherwise:

[
A_i \leftarrow min\left(1, A_i + \frac{\Delta T}{T_{age}}\right)
]

Suggested:

```text
T_age = 30000 ms
```

Interpretation:

| (A_i) | Meaning          |
| ----- | ---------------- |
| 0     | Recently updated |
| 1     | Very stale       |

---

## 9.4 Uncertainty Score (U_i)

This is not final probabilistic uncertainty.

This is **operational scan uncertainty**.

It measures how much the cell still needs useful sampling.

[
U_i = 0.6(1 - V_i) + 0.4A_i
]

Interpretation:

| Condition             | (U_i)       |
| --------------------- | ----------- |
| Never visited         | High        |
| Recently well sampled | Low         |
| Visited long ago      | Medium/high |
| Partially sampled     | Medium      |

---

## 9.5 Peakness Score (P_i)

Peakness measures whether a cell is locally stronger than its neighbors.

[
P_i = max\left(0, E_i - mean(E_j), j \in \mathcal{N}(i)\right)
]

Where (\mathcal{N}(i)) is the 4-neighbor or 8-neighbor set.

Recommended:

```text
Use 8-neighbor local contrast.
```

Interpretation:

| Case                                           | (P_i) |
| ---------------------------------------------- | ----- |
| Whole region has similar evidence              | Low   |
| One cell is clearly stronger than nearby cells | High  |
| Cell has no evidence                           | Low   |

---

## 9.6 Travel Cost (D_i)

Travel cost prevents inefficient target jumps.

Distance component:

[
d_i = \frac{distance(current_position, cell_center_i)}{d_{max}}
]

Heading component:

[
h_i = \frac{|\Delta\theta_i|}{180}
]

Combined:

[
D_i = 0.75d_i + 0.25h_i
]

If heading is unreliable:

[
D_i = d_i
]

---

## 9.7 Oscillation Penalty (R_i)

This prevents the target from changing too frequently.

Simple version:

[
R_i =
\begin{cases}
0, & i = previous_target \
1, & i \neq previous_target
\end{cases}
]

Better version:

[
R_i = 0.5\cdot \mathbb{1}[i \neq previous_target] + 0.5\cdot \mathbb{1}[i \notin neighborhood(previous_target)]
]

Interpretation:

* Small penalty for changing target
* Larger penalty for jumping far away

---

# 10. Final Decision Score

## 10.1 MVP Single-Mode Formula

Use:

[
J_i = 0.35E_i + 0.30U_i + 0.20P_i - 0.15D_i - 0.05R_i
]

Meaning:

| Term  | Weight | Purpose                         |
| ----- | -----: | ------------------------------- |
| (E_i) |   0.35 | Go where raw evidence is useful |
| (U_i) |   0.30 | Cover uncertain areas           |
| (P_i) |   0.20 | Refine local peaks              |
| (D_i) |  -0.15 | Avoid wasteful travel           |
| (R_i) |  -0.05 | Avoid unstable guidance         |

The selected target is:

[
i^* = \arg\max_i J_i
]

---

# 11. Explore / Refine Modes

After MVP, add explicit modes.

## 11.1 Explore Mode

Used when the system needs broad coverage.

[
J_i^{explore}=0.20E_i + 0.50U_i + 0.10P_i - 0.15D_i - 0.05R_i
]

Explore prioritizes:

* Unvisited cells
* Stale cells
* General area coverage

## 11.2 Refine Mode

Used when a likely peak region appears.

[
J_i^{refine}=0.45E_i + 0.15U_i + 0.30P_i - 0.07D_i - 0.03R_i
]

Refine prioritizes:

* Cells near strong evidence
* Local contrast
* More samples near suspected peaks

## 11.3 Enter Refine Mode

Enter Refine if a cell (k) satisfies:

```text
E_k > theta_E
P_k > theta_P
peak persists for M consecutive windows
```

Suggested defaults:

```text
theta_E = 0.65
theta_P = 0.20
M = 2 or 3 windows
```

## 11.4 Exit Refine Mode

Exit Refine if:

```text
peak disappears
OR local region is sufficiently covered
OR max refine duration exceeded
OR another peak becomes significantly stronger
```

Suggested:

```text
max_refine_duration = 20–30 seconds
```

# 19. Acceptance Criteria

## Backend

The feature is acceptable if:

* The Ground Station can create a grid from a predefined search area.
* The Ground Station can ingest PosePacket and EvidencePacket.
* Each cell maintains evidence, coverage, uncertainty, peakness, and final score.
* The system computes a target cell every 3–5 seconds.
* The system avoids rapid target oscillation.
* The system handles stale GPS or stale evidence safely.
* The system logs recommendation history.

## Frontend

The UI is acceptable if:

* The pilot sees the drone location.
* The pilot sees the current target cell.
* The pilot sees direction and distance to the target.
* The pilot sees Explore/Refine mode.
* The pilot sees whether the recommendation is stale or valid.

## Pi-side

The Pi-side behavior is acceptable if:

* It does not stream raw CSV live.
* It does not send full PCAP live.
* It sends bounded-size messages.
* It can survive bursty packet environments.
* It keeps local raw logs for the existing pipeline.

guidance

# 21. Final Recommended Design

Use this as the actual MVP design:

## Pi Sends

1. `PosePacketV1` at 2Hz
2. `EvidencePacketV1` every 1–2 seconds
3. Optional `HealthPacketV1` every 5–10 seconds

## Ground Station Computes

For each grid cell:
